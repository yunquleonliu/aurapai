**Aura-PAI 平台后端软件设计文档 (SDD)**
=======================================

**1. 引言**
-----------

### **1.1. 目的**

本文档旨在为"Auro-PAI 平台"的后端服务提供全面的软件设计规范。Auro-PAI
旨在通过集成本地大型语言模型（LLM）、本地向量数据库和外部互联网访问能力，为"普通人"（Average
Joys, AJs）提供上下文感知、透明且可控的AI协助，其中强大的代码增强功能是其可选但重要的组成部分。本文档将作为构建后端服务的指导蓝图，涵盖架构、模块设计、功能实现、数据管理和部署策略。

### **1.2. 范围**

本 SDD 专注于 Auro-PAI 平台后端服务的设计与实现。它详细说明了核心
FastAPI 应用程序，该应用程序将充当中央协调器，负责处理客户端请求，管理与各种
LLM（特别是通过 llama.cpp 运行的 LLaVA + Mixtral ），集成 RAG 系统
(ChromaDB)，执行外部工具（例如网络搜索和 URL
获取），以及管理会话上下文。

### **1.3. 目标与非目标**

**目标：**

-   构建一个健壮的 FastAPI 后端，作为 Auro-PAI 功能的核心。

-   实现与本地 llama.cpp LLM 服务器（托管 LLaVA + Mixtral 8x7B GGUF 模型）的无缝集成。

-   开发检索增强生成 (RAG) 管道，利用本地代码库和文档进行上下文检索。

-   集成 LLM 驱动的工具使用功能（网络搜索、URL 获取），使 AI 能够智能地获取外部信息。

-   确保 AI 建议（如代码重构）是透明的，并以结构化、可审查的格式呈现。

-   支持多种 LLM 提供商（本地、OpenAI、Gemini）以提供灵活性。

**非目标：**

-   本文档不涵盖前端（例如 VS Code 扩展）的详细设计。

-   不涉及 llama.cpp 或 ChromaDB本身内部的修改。它们被视为外部依赖或服务。

-   不涉及复杂的认证和用户管理，假定为单用户或信任环境。

### **1.4. 目标受众**

- 普通用户，对 提示词工程，软件开发，或者大模型训练没有兴趣，也没有相关背景知识。

- One that care their privacy enought to pay a dim, in onyway or another.

**3. 当前实现与主要特性总结**
--------------------------

本节总结了当前 Auro-PAI 后端的主要实现特性，并与设计目标进行对比：

### 3.1 已实现的主要功能

- **本地 LLM 集成**：支持 llama.cpp（LLaVA + Mixtral 8x7B GGUF），并可扩展至 OpenAI、Gemini。
- **RAG 检索增强生成**：集成 ChromaDB，支持本地代码库/文档的分块、索引与语义检索。
- **多模态能力**：支持文本与图片生成（如通过 prompt 生成图片、菜单图片生成菜品图像）。
- **Web 搜索工具**：集成 DuckDuckGo 搜索，支持实时互联网信息检索，并将结果注入 LLM 上下文。
- **URL 内容抓取**：可抓取并提取指定网页内容，辅助 LLM 回答。
- **会话与上下文管理**：支持多会话、历史消息管理、上下文截断与内存优化。
- **健康检查与监控**：提供各服务健康检查、就绪性/liveness 探针。
- **API 设计**：RESTful 风格，分层路由，支持流式响应与多种输入输出格式。
- **前端集成**：前端支持文本、图片、流式消息渲染。

### 3.1.1 典型AI能力与工具特性列表

本平台后端已实现并支持如下典型AI能力与工具特性，所有能力均可通过 LLM 以 JSON 工具调用协议自动编排：

- **search_and_fetch**：互联网检索与摘要（如 DuckDuckGo 实时搜索，自动抓取网页内容并摘要）。
- **image_generation**：根据文本描述生成图片。
- **web_search**：实时互联网搜索，返回摘要或原始内容。
- **url_fetch**：抓取指定网页内容，结构化注入 LLM 上下文。
- **text_generation**：上下文感知的文本生成、总结、改写等。
- **image_analysis**：图片内容分析、OCR、视觉问答等。
- **clarification**：意图不明时自动澄清，提升交互鲁棒性。

> 以上能力均通过 LLM 工具调用协议（如 {"action": "search_and_fetch", "query": "..."}）实现，后端自动解析、执行并将结果注入 LLM，形成“LLM-工具-LLM”闭环。

### 3.2 主要模块说明

- `main.py`：FastAPI 应用主入口，初始化各服务与路由。
- `api/routes/chat.py`：主聊天接口，意图检测，路由到 LLM、图片生成或澄清，构建上下文（RAG、Web 搜索、历史）。
- `api/routes/tools.py`：Web 搜索、URL 抓取等工具接口。
- `services/rag_service.py`：ChromaDB 检索、索引、上下文获取。
- `services/tool_service.py`：Web 搜索（DuckDuckGo）、URL 抓取、工具注册。
- `services/image_generation_service.py`：图片生成逻辑。
- `core/context_manager.py`：会话与上下文管理。
- `static/js/app.js`：前端聊天逻辑、图片渲染、流式处理。

### 3.3 与设计目标的对比

- 绝大部分设计目标已实现，包括本地 LLM、RAG、Web 搜索、工具编排、多会话、健康检查等。
- 新增了多模态（图片生成/分析）能力，超出原始设计。
- 工具与 RAG 的集成较为灵活，但多工具协作和错误处理仍有提升空间。
- 某些功能（如 Web 搜索与图片生成）在集成时存在回归风险，需进一步解耦和测试。

**8. 查询-意图-路由-输出一致性工作流（2025年6月）**
--------------------------------------

本节明确 Auro-PAI 后端的核心工作流，确保“用户查询-意图识别-服务路由-输出类型”全链路一致：

### 8.1 工作流步骤

1. **用户查询接收**
    - 用户通过 Web 前端（或未来多端）提交文本或图片。

2. **意图检测（Intent Detection）**
    - 后端调用 LLM 对查询进行意图分类：
        - `image_generation`（如“生成一张鸡的图片”）
        - `image_analysis`（如“请读图”+图片）
        - `web_search`（如“最新新闻”）
        - `text_generation`（如“总结这段代码”）
        - `ambiguous`（不明确时要求澄清）

3. **严格路由**
    - `image_generation`：调用图片生成服务，返回图片（不回退为文本，除非生成失败）。
    - `image_analysis`：调用视觉/LLM分析，返回文本。
    - `web_search`：调用 Web 搜索，结果注入 LLM 上下文，返回文本。
    - `text_generation`：构建上下文（历史+RAG+Web 搜索），调用 LLM，返回文本。
    - `ambiguous`：返回澄清提示。

4. **输出类型与意图绑定**
    - 图片请求始终返回图片，文本请求始终返回文本。
    - 响应结构明确区分类型，前端据此渲染。

5. **前端渲染**
    - 图片用 <img> 渲染，文本用消息气泡渲染。
    - 保证用户体验与意图一致。

### 8.2 设计原则

- **意图与输出强绑定**：如“生成图片”只返回图片，“读图”只返回文本。
- **后端为“意图路由器”**：不做“糊涂”回退，除非服务失败。
- **前后端协作**：前端根据响应类型渲染，避免误导用户。
- **可扩展性**：未来可支持更多意图类型（如代码修复、表格识别等）。

> 该工作流确保平台行为可预测、体验一致，便于维护和扩展。

### 8.3 互联网增强RAG（search-and-fetch）工作流

本节详细描述 Auro-PAI 后端支持的互联网增强型 RAG（Retrieval-Augmented Generation）工作流，支持 LLM 自动调用外部工具（如 Web 搜索、网页抓取、图片生成等），实现“LLM-工具-LLM”闭环。

#### 工作流步骤

1. **用户查询**：用户通过前端提交问题（如“2025年中国AI行业最新动态？”）。
2. **LLM 工具调用输出**：后端 prompt LLM，要求其以 JSON 工具调用格式输出（如 {"action": "search_and_fetch", "query": "2025年中国AI行业最新动态"}）。
3. **后端解析与工具执行**：后端解析 LLM 输出，自动调用对应工具（如 DuckDuckGo 搜索、网页抓取、图片生成等）。
4. **结果注入与二次 LLM 归纳**：工具结果（如网页摘要、图片链接）被结构化注入 LLM 上下文，后端再次 prompt LLM 进行归纳、总结或多轮推理。
5. **最终响应**：LLM 输出最终答案，后端返回给用户。

#### 关键要点

- LLM 工具调用协议采用结构化 JSON，便于解析与扩展。
- 后端支持多工具协同（如先搜索再抓取网页，再由 LLM 归纳）。
- 支持多轮“LLM-工具-LLM”循环，直到获得最终答案。
- 工具调用与 LLM 推理解耦，便于后续扩展更多工具。
- 典型用例包括：互联网实时问答、网页内容总结、图片生成与分析等。

#### 典型用例


- “帮我查下2025年中国AI行业最新动态” → LLM 输出 search_and_fetch → 后端搜索并抓取网页 → LLM 归纳 → 返回摘要。
- “生成一张未来城市的图片” → LLM 输出 image_generation → 后端生成图片 → 返回图片链接或 base64。
- “请分析这张图片内容” → LLM 输出 image_analysis → 后端分析图片 → LLM 归纳 → 返回分析结果。

**9. 会话数据存储与用户隐私设计**
-----------------------------

### 9.1 聊天历史与数据存储（统一用 ChromaDB）

- 当前实现：
    - 会话历史（chat history）主要存储在后端内存（ContextManager），用于短期上下文管理。
    - RAG（检索增强生成）相关的代码/文档分块、向量等存储在 ChromaDB。
- 设计优化：
    - **统一用 ChromaDB 存储用户会话历史**，避免引入多种数据库，简化架构。
    - 每条消息作为一个文档存入 ChromaDB，包含：用户ID/会话ID、消息内容、时间戳、消息类型（文本/图片/系统等）。
    - 支持按用户/会话/时间/关键词检索历史。
    - RAG 仅存储知识分块，用户历史与知识库分开管理但同用 ChromaDB。

### 9.2 用户历史访问与管理

- 用户可通过 API/前端查看自己的历史会话
- 支持历史检索、删除、导出等操作。
- 可按会话、时间、关键词等多维度检索。

### 9.3 用户隐私保护方案

- **最小化存储原则**：仅存储必要的会话数据，RAG/向量库不存储用户私聊内容。
- **本地优先**：默认数据仅存储在本地服务器/数据库，不上传云端。
- **加密存储**：敏感数据（如用户ID、消息内容）可加密存储，防止泄露。
- **访问控制**：仅允许用户本人访问/管理自己的历史，后台管理需严格权限。
- **可删除性**：用户可随时删除自己的历史记录，系统应彻底清除。
- **合规性**：遵循 GDPR/中国个人信息保护法等相关法规，定期审计。

> 未来可扩展为多用户/多端同步，支持 OAuth 登录、端到端加密等更高级隐私保护。

**10. 前端聊天历史管理与隐私控制设计**
-----------------------------------

### 10.1 历史记录访问与管理 UI 方案

- **统一体验**：Web 端与未来手机端（如 Android/iOS）采用一致的聊天历史管理界面，降低用户学习成本。
- **混合式布局**：
    - 桌面端优先采用“侧边栏+主区”布局：
        - 左侧侧边栏展示会话列表（按时间/主题分组，可滚动）。
        - 右侧主区显示当前会话消息。
        - 用户无需切换页面即可在会话间快速切换。
    - 移动端/小屏幕自动切换为“历史 Tab”模式：
        - 底部或顶部 Tab，点击“历史”进入历史会话列表。
        - 选中会话后进入消息详情，可返回列表。
    - 响应式设计，自动适配屏幕尺寸，保证一致体验。

### 10.2 搜索与筛选

- 支持按关键词、时间范围、会话分组等多维度检索历史消息。
- 搜索框与筛选器固定在侧边栏/历史 Tab 顶部，便于随时检索。

### 10.3 消息与会话操作

- 每条消息/会话支持：
    - 删除（带确认提示）
    - 导出（JSON/CSV 文件下载）
- 支持“一键清空全部历史”操作。

### 10.4 隐私与合规提示

- 明确告知“所有历史仅本地存储，可随时删除”。
- 显示隐私政策入口。
- 删除操作彻底清除后端数据。

### 10.5 API 对接与前端实现建议

- 前端通过 `/chat/history`、`/chat/history/export`、`/chat/history/delete` 等 API 获取和管理历史。
- 页面加载或用户点击历史入口时自动拉取会话列表。
- 响应式布局，桌面端侧边栏+主区，移动端 Tab 切换。
- 所有操作（删除/导出）均有反馈，操作后自动刷新。

> 该设计兼顾桌面与移动端体验，便于未来多端统一维护和扩展。
